# Архитектура ebay_library

Библиотека методов парсинга eBay для воркеров. Что парсим и почему — здесь и в
flow-спеках; селекторы — в [catalog_selectors.md](catalog_selectors.md) /
[item_selectors.md](item_selectors.md) и единственной точкой в коде
`ebay_library/parsing/selectors.py`.

---

## Границы (in / out of scope)

**In** — всё, что воркер вызывает как метод: построение URL выдачи (с фильтрами
ZIP/состояние/цена), детект страницы, ожидание готовности, навигация по
выдаче/товару, парсинг, скачивание фото, объединение блока подзадач.

**Out** — оркестрация: пул воркеров и их запуск, прокси/стелс, раздача задач,
ретраи упавших подзадач, передача состояния умершего воркера, запись в БД.
Воркер получает снаружи готовый Playwright `page` — библиотека его не создаёт
и не закрывает.

Отложено (обсудим отдельно): кэш и фильтрация трафика
([traffic_filtering.md](traffic_filtering.md)), скачивание фото, «написать
продавцу», возобновляемость после смерти воркера.

## Два слоя

### Слой 1 — парсинг-ядро (чистые функции, без браузера)

`HTML/url → данные`. Тестируется на сохранённых фикстурах (`tests/fixtures/`).

| модуль | владеет |
|---|---|
| `page_state.py` | «что за страница» (url+title, БЕЗ DOM) + antibot |
| `catalog.py` | разбор страницы выдачи → `SearchPage` |
| `item.py` | разбор страницы товара (2 HTML) → `ItemPage` |
| `normalize.py` | общие нормализации (condition new/other) — один источник для каталога и item |
| `selectors.py` | ВСЕ CSS-селекторы (единственное место, DRY) |
| `models.py` | dataclasses (frozen, slots) |
| `config.py` | построение URL выдачи (`build_search_url`, `ITEMS_PER_PAGE`) — фильтры zip/condition/цена |
| `download.py` | скачивание фото (`fetch_images`, httpx, без браузера) |

### Слой 2 — методы воркера (живой Playwright `page`)

| модуль | владеет |
|---|---|
| `navigation.py` | ожидание готовности (`wait_until_ready`) + политики блоков |
| `fetch.py` | `warmup`, `fetch_item`, `fetch_catalog`, `fetch_catalogs` |

## Жизненный цикл воркера

1. **Старт — один раз**: `warmup(page)` — заход на главную, ожидание
   готовности. Без прогрева первый SRP ловит антифрод (Error Page / Pardon).
   Дальше на главную не возвращаемся.
2. **Блоки подзадач**: `fetch_catalogs(page, queries, zip=…, condition=…, …)` —
   каталоги по списку запросов, фильтры (ZIP/состояние/цена) задаются через URL.
   Объединённый результат без дублей. Упавшая подзадача не валит блок
   (`errors`), её переотдаст оркестратор.
3. **Товары**: `fetch_item(page, item_id)` — по отдельной задаче (после
   каталога items автоматически НЕ парсим). Фото — `fetch_images(urls)`.
4. Контекст между задачами не закрывается. ZIP не ставится через UI — он
   параметр URL (`_stpos`), применяется на каждой странице выдачи.

## Философия ошибок

Обязательное поле не выбили → `ParseError` с сырьём (HTML карточки/страницы)
наружу — без тихих фолбеков и подстановки None. Опциональны только поля,
для которых живьём подтверждено легитимное отсутствие: `last_updated` (item),
`location` (каталог). Antibot-блоки → исключения по политикам из
[page_readiness.md](page_readiness.md). В batch ошибки изолируются
per-подзадача.
