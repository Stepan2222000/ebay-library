# Архитектура ebay_library (v2 — пакет `ebaylib`)

Библиотека парсинга eBay для воркеров: каталог (SRP), товары (PDP), фото,
запись результатов в БД `ebay_data`, цикл воркера. Что парсим и почему — здесь
и в flow-спеках; селекторы — в [catalog_selectors.md](catalog_selectors.md) /
[item_selectors.md](item_selectors.md) и единственной точкой в коде
`ebaylib/html/selectors.py`.

---

## Границы (in / out of scope)

**In** — всё, что выполняет воркер: парсинг выдачи/товара на живом Playwright
`page` (с заменой страниц при блокировках), конвертация цен в USD, скачивание
фото, запись результатов в `ebay_data` (клиент серверного API), цикл
«задача → парсинг → запись → подтверждение».

**Out** — оркестрация: пул воркеров и их запуск, браузер/прокси/стелс,
очередь задач и переотдача невыполненных, S3-заливка фото. Воркер отдаёт
библиотеке три колбека (`get_page`, `next_task`, `task_done`) и DSN — больше
ничего своего у него нет.

**Хранение** — тоже не наше: вся логика БД (апсерты, диффы, журнал изменений,
death/resurrection) живёт в серверных функциях проекта `ebay_data`
(`db/schema.sql` там); библиотека лишь сериализует результат в jsonb и зовёт
`apply_catalog_fetch` / `apply_item_snapshot`.

---

## Слои (раскладка пакета)

```
ebaylib/
  errors.py    исключения (чистые, без playwright)
  models.py    dataclasses результатов — контракт API
  urls.py      build_search_url, ITEMS_PER_PAGE (фильтры zip/condition/цена)
  html/        Слой 1: чистый парсинг HTML/url → данные (без браузера и сети)
    page_state.py   «что за страница» (url+title, БЕЗ DOM) + antibot
    srp.py          разбор страницы выдачи → SearchPage
    item.py         разбор товара (2 HTML) → ItemPage; ship_to_location (ZIP)
    normalize.py    нормализация condition — один источник для SRP и PDP
    selectors.py    ВСЕ CSS-селекторы (единственное место, DRY)
  http/        HTTP-IO без браузера
    fx.py           конвертация валют каталога в USD (fx-микросервис)
    images.py       скачивание фото (CDN i.ebayimg.com, без антибота)
  browser/     Слой 2: живой Playwright
    readiness.py    ожидание готовности (wait_until_ready) + политики блоков
    session.py      EbaySession — добыча с контуром замены страниц
  store.py     запись в ebay_data (asyncpg-клиент серверных apply_*-функций)
  worker.py    run_worker — цикл «задача → парсинг → запись → task_done»
```

Слой 1 тестируется на фикстурах (`tests/fixtures/`) без браузера; Слой 2 —
на фейковых page (`tests/test_session.py`, `test_worker.py`) и живых smoke.

## Публичное API — только необходимое

```python
from ebaylib import (
    run_worker, Store, EbaySession,          # точка входа воркера
    fetch_images,                            # фото (отдельная операция)
    CatalogResult, Catalog, CatalogItem, ItemPage,   # результаты
    ParseError, AccessDeniedError, ErrorPageError, TaskFormatError,  # смерти
)
```

Внутренности (`ebaylib.html.*`, `ebaylib.urls`, `ebaylib.http.*`,
`ebaylib.browser.readiness`) доступны по полным путям для тестов/отладки,
но публичным контрактом не являются.

## EbaySession: страницы и контур замены

Сессия не получает готовый `page` — она получает **`get_page`** (async-колбек
воркера «дай свежую страницу»; каждый вызов обязан отдавать новый
контекст/прокси, старые страницы утилизирует воркер) и сама:

- лениво берёт первую страницу при первой надобности и прогревает её
  (заход на главную; без прогрева первый SRP ловит антифрод);
- при **блокировке eBay** (Access Denied) или **транспортной смерти**
  страницы (`net::ERR_*`, краш, закрытие) выдерживает фикс. паузу
  `page_delay_s` (дефолт `PAGE_DELAY_S`), просит новую страницу, прогревает
  и продолжает **с того же места**: каталог — с упавшей страницы выдачи
  (собранное не теряется), item — товар целиком. Лимита замен нет — темп и
  счёт попыток контролирует воркер внутри `get_page`.

Пока страница жива, `get_page` не вызывается — сколько угодно задач подряд
идут на одной странице.

## Философия ошибок

Заменой страницы лечатся ТОЛЬКО блокировка eBay и транспортная смерть.
Всё остальное — **критические ошибки**: летят наружу и валят задачу и воркера
целиком, без фолбеков и ретраев:

| Ситуация | Действие |
|---|---|
| Pardon | ждём до 180с; не прошёл → TimeoutError (критично) |
| «Error Page \| eBay» | ErrorPageError (критично) |
| Access Denied | замена страницы, продолжение с того же места, без лимита |
| net::ERR_* / краш / закрытие page | то же — замена страницы |
| ParseError, сбой fx, таймауты якорей/iframe, кривая задача | критично |

Обязательное поле не выбили → `ParseError` с сырьём (HTML) наружу — без
тихих фолбеков. Опциональны только поля с подтверждённым live легитимным
отсутствием: SRP — `condition`, `location`, `shipping_cost` (None = «не
указана»/Freight/самовывоз); PDP — `last_updated`, `condition` («--» not
specified), `shipping_cost` («contact seller»). Машинерии «мёртвой сессии»
нет: после критического исключения воркер завершается, объект сессии умирает
с ним.

## Запись: Store (клиент серверного API ebay_data)

`Store(dsn)` — одно asyncpg-соединение (лениво), `statement_cache_size=0`
(pgbouncer-ready: при появлении bouncer меняется только DSN). DSN захардкожен
дефолтом (паттерн fx), env `EBAY_DATA_DSN` переопределяет.

- `apply_catalog(article, catalog, zip=…, condition=…, min/max_price=…)` →
  `apply_catalog_fetch` — один артикул за вызов (транзакция на артикул);
  фильтры задают search-профиль в БД; карточкам добавляется
  `"currency": "USD"`. Возвращает статистику
  `{fetch_id, items_total, items_new, appeared, missed, deactivated, died}`.
- `apply_item(item, zip=…)` → `apply_item_snapshot` → `{item_id, is_new,
  specifics, images}`.

Артикул обязан существовать в smart-каталоге (`smart_fdw.part_articles`) —
иначе функция бросает (критично, защита от опечаток оркестратора). Повторная
запись идемпотентна (нулевые диффы) — переотдача задач безопасна.

## run_worker: цикл воркера

```python
await run_worker(get_page, next_task, Store(), task_done=task_done)
```

Pull-модель: цикл сам зовёт `next_task()` после каждой записанной задачи;
нет задач — колбек просто ждёт; вернул `None` — штатное завершение. Формат
задачи (всё вне `params` — метаданные оркестратора, едут в `task_done` как
есть; кривой формат → `TaskFormatError`, критично):

```json
{"type": "catalog", "params": {"articles": ["805079T", "805079"],
                               "zip": "19701", "condition": "new",
                               "min_price": 50, "max_price": 500}}
{"type": "item",    "params": {"item_id": "277574984378", "zip": "19701"}}
```

Парсинг и запись перекрыты (ограниченная очередь + фоновый писатель), но
контракт жёсткий: **задача = парсинг + запись; `task_done(task, stats)`
вызывается строго ПОСЛЕ записи** («обработана = записана»). Ошибка парсинга →
хвост очереди дописывается, затем смерть; ошибка записи → смерть сразу (даже
если парсинг ждёт задач). Незаписанные задачи (без `task_done`) переотдаёт
оркестратор.

## Диагностика

Стандартный `logging`, логгер `"ebaylib"`: INFO — замены страниц (причина и
место), DEBUG — прогресс (прогрев, страницы выдачи, записи). Без хендлера
у воркера — библиотека молчит.
