# Фильтрация трафика в Playwright/CloakBrowser

Скрапинг через прокси упирается в одно: байты дорогие. Голая страница eBay SRP
тянет несколько МБ декодированного HTML и ещё больше — картинок, скриптов,
трекеров. Без фильтрации это десятки гигабайт на проект. Эта спецификация
описывает, что блокировать, что кэшировать и что трогать нельзя.

---

## TL;DR

- **Блокируем целиком**: `image`, `font`, `media`, плюс хосты трекеров и
  тяжёлые внутренние `xhr`, которые не дают данных для парсера
  (recommendations, analytics).
- **Кэшируем на диск**: `script`, `stylesheet` — они общие между страницами
  крупных сайтов, после первого визита ловятся в кэш почти всегда.
- **НЕ кэшируем**: `document` (это наш парс-результат, уникальный на каждый
  артикул), `xhr/fetch` с динамическими токенами и личной сессией.
- **Никогда не блокируем**: антибот-челленджи (`splashui/challenge`,
  `argon2.wasm`, активные fingerprint-сборщики типа PerfDrive). Иначе
  следующий запрос вернётся как страница блокировки.

---

## Где живёт кэш

Конкретный путь зависит от проекта — у нас это просто папка на диске рядом со
скриптом. В этой папке:

- файл с сырым телом ответа (raw bytes, уже разжатое);
- сосед — JSON-метаданные (status + headers) для того же ключа.

Ключ — это `sha1(method + url)` от исходного запроса. То есть тот же скрипт
по тому же URL даёт один и тот же файл, кэш быстро сходится между
параллельными воркерами.

## Как Playwright перехватывает запросы

`page.route(pattern, handler)` ставит обработчик на каждый запрос со страницы
(включая запросы из iframe'ов, `fetch()`, скриптов). В handler'е доступны 4
действия:

- **`route.abort()`** — убить запрос. Подходит для всего, что нам не нужно
  (картинки, шрифты, реклама).
- **`route.continue_()`** — пропустить как есть. Стандарт для документа и
  полезных xhr.
- **`route.fulfill(status, headers, body)`** — подделать ответ. Так мы
  отдаём данные из локального кэша.
- **`route.fetch()`** — скачать самим, тем же сетевым стеком, что и контекст
  (тот же proxy и TLS). Результат потом отдаём через `fulfill`.

Подробности в документации:
[Page.route](https://playwright.dev/python/docs/api/class-page#page-route),
[Route](https://playwright.dev/python/docs/api/class-route),
[Request.resource_type](https://playwright.dev/python/docs/api/class-request#request-resource-type).

## Что блокировать

### Графика и шрифты

Это самая жирная часть страницы. Парсер карточек читает только текст и
атрибуты в HTML, картинки для него не нужны.

```python
BLOCK_TYPES = {"image", "font", "media"}
```

HTML с карточками всё равно приходит, у `<img>` остаётся атрибут `src=` —
парсер с ним работает. eBay JS пытается lazy-load изображение, получает
ошибку загрузки и тихо игнорирует.

Сломается только сценарий со скриншотом или визуальным
fingerprint-чеком — для нас нерелевантно.

### Трекеры по hostname

Эти домены живут только для аналитики и рекламы. Сайт работает без них:

```python
BLOCK_HOSTS = {
    "gum.criteo.com",
    "ct.pinterest.com",
    "fls.doubleclick.net",
    "securepubads.g.doubleclick.net",
    "tr.snapchat.com",
    "sherlock-events-sampler.ebay.com",
    "13341218.fls.doubleclick.net",
}
```

### Тяжёлые внутренние xhr по URL-паттерну

`useracquisition` на eBay — самый тяжёлый из бесполезных для парсера xhr
(сотни КБ декода на каждую SRP, персональные рекомендации). Эндпоинт
подгружается JS-ом уже после `domcontentloaded`, к этому моменту SRP
отрисована и парсер работает. JS получит ошибку и тихо проглотит.

```python
BLOCK_URL_PATTERNS = (
    "/gh/useracquisition",
    "/sch/ajax-service-bridge",
    "/sch/ajax/autocomplete",
)
```

## Что кэшировать

### Скрипты и стили

JS и CSS у крупных сайтов общие между страницами с очень редкими
обновлениями. После одного полного визита кэш насыщается, и дальше
запросы за этими ресурсами уходят в локальный диск, минуя прокси.

Алгоритм коротко:

1. На вход прилетает запрос с `resource_type in {"script", "stylesheet"}`.
2. Считаем `sha1(method + url)` — это ключ.
3. Есть на диске → `route.fulfill(...)` из кэша.
4. Нет → `route.fetch()`, сохраняем тело + статус + headers, отдаём
   через `route.fulfill(...)`.

Подводный камень — `Content-Encoding`. `route.fetch()` возвращает уже
распакованное тело, но в headers стоит `Content-Encoding: gzip` и
`Content-Length: <gzip-size>` от оригинала. Браузер попробует распаковать ещё
раз и упадёт. Вырезаем эти заголовки перед `fulfill`:

```python
STRIP_HEADERS = {"content-encoding", "content-length", "transfer-encoding"}
```

### Документ

`document` — это сам HTML страницы. **Не кэшируем**.

### `xhr` и `fetch`

Часто несут токены сессии, timestamps, флаги A/B. Если отдать «вчерашний»
ответ — антибот заметит несоответствие сессии и зарежет. По умолчанию **не
кэшируем**.

## Что трогать нельзя

Иначе антибот срабатывает (Pardon Our Interruption / Access Denied):

- `*.ebaystatic.com/cr/.../argon2.wasm` — криптография для PerfDrive
  challenge.
- `cas.avalon.perfdrive.com/jsdata` — активный антибот-сборщик.
- `www.ebay.com/splashui/challenge` — сам челлендж-страница.
- Любой inline-bootstrap скрипт страницы, без которого render не доходит до
  конца.

Эмпирическое правило: если домен принадлежит самому сайту (`*.ebay.com`,
`*.ebaystatic.com`, `*.ebayimg.com`) и приходит **до** `domcontentloaded` —
не блокируем. Тяжёлые xhr **после** `domcontentloaded` (как
`useracquisition`) — можно.

## Универсальный чек-лист

Принципы те же для любого сайта:

1. Запусти один прогон без фильтра, сохраняя все НЕ-заблокированные ответы
   (или просто всё подряд).
2. Посчитай `seen_count × body_size` по уникальным URL и hostname'ам.
3. Идентифицируй три группы:
   - тяжёлые и бесполезные для парсера → блокировать;
   - тяжёлые и общие между страницами → кэшировать;
   - тяжёлые и уникальные (документ, ваш парс-таргет) → не трогать.
4. Найди антибот-эндпоинты (challenge JS, fingerprint API) и **внеси в
   whitelist**.
5. Прогон → новый дамп → итерация.
