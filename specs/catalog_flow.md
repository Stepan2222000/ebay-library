## URL выдачи и фильтры

URL строит `ebaylib.urls.build_search_url(query, *, page, zip, condition,
min_price, max_price)`. Структурные части всегда: `_nkw` (запрос,
URL-энкодится), `_ipg=240` (`ITEMS_PER_PAGE`, максимум eBay), `_pgn`
(страница, 1-based), `_dmd=1` (режим отображения). Фильтры опциональны —
добавляются только если заданы:

⚠️ `_dmd=1` обязателен (форсит **list-view**). eBay A/B показывает SRP двумя
вёрстками: **list-view** (список) — у карточек есть строка продавца
`<ник> NN% positive`; **grid-view** (сетка) — продавца НЕТ нигде (ни в DOM,
ни в embedded-JSON). Без `_dmd` браузер случайно попадает в grid-view, и
парсер падает на обязательном `seller=None` (live 2026-06-13: запросы
0914930/65606/854553 в сетке — все карточки без продавца; с `_dmd=1` —
список с продавцом, парсятся). Не путать с lazy-рендером: в grid-view
продавца нет физически.

- `zip` → `_stpos`: индекс доставки. **Задаёт РЕАЛЬНУЮ локацию**, не косметику:
  доставка пересчитывается под ZIP (live: 19701 vs Аляска 99950 → разные суммы
  доставки и число результатов). Без ZIP у многих карточек доставка не
  рендерится ⇒ на практике **ZIP нужен всегда** (рекоменд. "19701").
  `_fcid=1` (страна US) не обязателен — `_stpos` достаточно.
- `condition` → `LH_ItemCondition`: `"all"`/`None` — без фильтра; `"new"` = `3`
  (Brand New + New Other); `"used"` = `3000` (Used / Pre-Owned). Фактическое
  состояние карточки всё равно парсим.
- `min_price`/`max_price` → `_udlo`/`_udhi`: границы цены (USD). Подтверждено
  live, фильтруют реально.

ZIP — это URL-параметр, **UI-установки ZIP больше нет** (старый модал-флоу
удалён). Прежнее «`_stpos` косметический» неверно для текущего eBay.

Замечание: вёрстка SRP и валюта зависят от **страны IP** (US-IP → каноническая
вёрстка + USD; не-US IP → локализованная вёрстка и местная валюта, URL-параметром
не переопределяется). Воркер ходит через US-прокси.

## Парсинг страницы (Слой 1, `html/srp.py::parse_search_page`)

Возвращает `SearchPage` с `SrpCard` — сырьём «как на сайте» (цена/доставка в
исходной валюте + `currency_raw`), БЕЗ перевода в USD (это делает fx-шаг
сессии). Можно гонять офлайн на сохранённом HTML.

- Карточки до сепаратора «Results matching fewer words» — только точные;
  после — «похожие», не берём. Сепаратор встретился → `has_fewer_words_sep`.
- Placeholder «Shop on eBay» (data-listingid длиннее 12 цифр) — пропускаем.
- `results_count` — из счётчика; справочный (циклом пагинации не управляет).
- «0 results» — валидный пустой результат, не ошибка.
- Опциональные поля карточки (подтверждено live, → None):
  - `location` — «Located in» рендерится лениво и не всегда;
  - `condition` — у части карточек subtitle без состояния (напр. 400448473243);
  - `shipping_cost` — «Shipping not specified», «Freight» (грузовая без суммы
    на выдаче), самовывоз-only.
- `seller` ОБЯЗАТЕЛЕН → ParseError. Известное ограничение: в категориях
  Authenticity Guarantee (часы, кроссовки) продавца нет на всей выдаче — такие
  каталоги падают целиком; встречаются и одиночные карточки без продавца
  (live 365556461065). Обработку добавим, когда категории понадобятся.

## Пагинация (`EbaySession.fetch_catalog`)

Цикл `_pgn=1,2,…` — стоп-сигналы, в порядке проверки:

1. **сепаратор fewer-words** — точные кончились;
2. **страница неполная** (< `ITEMS_PER_PAGE` карточек) — последняя;
3. **`max_pages`** (дефолт `MAX_PAGES = 5`) — кап для широких запросов.

⚠️ eBay **за концом не отдаёт пустую страницу** — клампит на последнюю,
поэтому стоп «по 0 карточек» не работает; стоп по «<240» не делает лишнего
запроса. Кап страниц дружит с death-логикой БД через воскрешение: товар,
вновь попавший в выдачу, оживает любым наблюдением.

Между соседними страницами выдача **нахлёстывается** (хвост N ≈ начало N+1,
наблюдали 7–9 карточек) — дедуп по `item_id` обязателен (первое вхождение
побеждает). Счётчик может расходиться с фактом (±%) — свойство eBay.

Блокировка/смерть страницы посреди пагинации лечится сессией: замена страницы
и продолжение **с упавшей `_pgn`** — собранные карточки не теряются, дедуп
переживает замену.

## Конвертация цен в USD (`http/fx.py::convert_cards`)

Чистый парсер отдаёт `SrpCard` с ценой/доставкой в **исходной валюте** + сырым
токеном `currency_raw` (`'$'`, `'C $'`, `'EUR'`…). Перевод в USD — после
сбора всех страниц запроса, **один батч на каталог**: уникальные
`(сумма, валюта)` пары (дедуп) параллельно через
`GET {FX_API_URL}/convert?amount=&from=<токен>&to=USD`. `Free` (0.0) и
«не указана» (None) не конвертируются. Итог — `CatalogItem` в USD.

`FX_API_URL` — env, дефолт захардкожен. Без фолбеков: неизвестная валюта
(404) или недоступность сервиса → критическая ошибка, задача падает целиком.
Параллелизм без лимита — расчёт на воркеров рядом с сервисом (сервис держит
~200–250 одновременных соединений; с удалённой машины большие батчи падают —
проверено, решение принято осознанно).

## Универсальный вызов и результат

`fetch_catalog(queries, …)` принимает строку или список (группа алиасов
артикула); дубликаты запросов отбрасываются (на eBay не ходим дважды).
Результат `CatalogResult`:

- `items` — общий список с глобальным дедупом по item_id (первое вхождение
  побеждает);
- `per_query` — полный `Catalog` каждого запроса (дедуп только внутри
  запроса!) — именно он идёт в запись: товар, найденный по двум артикулам,
  должен попасть в membership обоих.

Поля `errors` нет: любой критический сбой валит вызов целиком, без частичных
результатов — переотдача задачи на оркестраторе.

## Запись (`Store.apply_catalog` → `apply_catalog_fetch`)

По вызову на артикул: payload = `Catalog` как jsonb (+ `"currency": "USD"`
карточкам). БД сама: резолвит контекст (zip) и search-профиль (condition/цены),
апсертит items/sellers/shipping/membership, ведёт журнал изменений
(триггеры), считает misses → деактивация → смерть (порог
`settings.death_threshold`), воскрешает при новом появлении. Повторная запись
идемпотентна. Подробности — `ebay_data/db/schema.sql`.
