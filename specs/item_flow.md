# Поток товара (item / PDP)

Селекторы полей — [item_selectors.md](item_selectors.md). Готовность/блоки —
[page_readiness.md](page_readiness.md). Здесь — контракт потока: два HTML,
правила полей, добыча, запись.

## Два HTML на один товар

Описание товара живёт в **отдельном iframe** (`iframe#desc_ifr` →
`itm.ebaydesc.com`) — в HTML основной страницы его текста НЕТ (видимый на
экране текст — отрендеренный iframe). Поэтому парсер принимает два документа:

```
html/item.py::parse_item_page(main_html, description_html=None) -> ItemPage
```

- `main_html` — все поля;
- `description_html` — `frame.content()` iframe-фрейма: передан → извлекаем
  текст (bs4, без script/style, `\n`-разделитель), не передан → `description=""`.

Документ фрейма **ленивый**: eBay начинает грузить его только при попадании
iframe во viewport (подтверждено live: без скролла не грузится вовсе), и
появляется он не сразу — сначала `about:blank`, затем реальный `ebaydesc.com`.
Добыча (`browser/session.py::_description_html`): скролл к `#desc_ifr`, затем
ждать фрейм с этим host + `domcontentloaded` (до 15с), нет → TimeoutError
(критично — медленный-но-живой прокси заменой страницы не лечим).

## Правила полей

- **seller — username**, не display-name. Единый источник — embedded-JSON
  страницы: ключ `"sellerUserName"` (ровно одно вхождение, всегда ник
  владельца листинга; проверено на 23 PDP: магазин/без магазина/EU-сессии).
  Ссылки карточки продавца НЕ годятся: их формат зависит от наличия магазина
  (`_ssn=` / `/str/` / `/sch/<user>/m.html`), слаг `/str/` может отличаться от
  ника (`avantims` ≠ `avantimotorsports`), а `/str/` ловит и ЧУЖИЕ магазины из
  рекламных блоков (live 376748192596: `/str/gpscity`). Формат ника совпадает
  с каталогом — общий ключ для сопоставления.
- **Цена — всегда USD**: есть `.x-price-approx` («Approximately US $X», intl)
  → берём его; нет (US-листинг) → primary. Суффикс `/ea`,`/lot` отбрасываем.
  Native-валюту не храним.
- **Доставка — итог в USD**: `Free…` → 0.0; intl → число из `(approx US $X)`;
  US → `US $X` в начале строки. **None** (опционально, live-кейсы): продавец
  не указал доставку до ZIP — eBay пишет «…contact seller for shipping
  options» без суммы (174601590686); самовывоз-only без строки доставки.
  Прочие непарсящиеся форматы → ParseError.
- **condition** — нормализация через общий `normalize.py` (тот же, что
  каталог): new/other. **None** (опционально): продавец не указал состояние —
  блок рендерится как «`--` not specified» (live 324023434393, марки;
  у коллекционных категорий вместо Condition свои поля качества).
  Неизвестная строка состояния → ParseError (словарь пополняем только
  пойманным живьём, напр. 'New with tags' 115851272608).
- **specifics** — вся таблица key→value, как есть.
- **image_urls** — большие версии фото. Единый способ: URL каждой картинки
  карусели (`src`; у ленивых картинок URL в `data-src`) + замена токена
  размера на `s-l1600` — CDN всегда отдаёт максимум существующего (клампит к
  оригиналу, не 404; проверено httpx). От `data-zoom-src` не зависим: у части
  листингов он ПУСТОЙ (большой версии нет физически, оригинал 500px — live
  125943066590). Дедуп с сохранением порядка карусели.
- **last_updated** — опциональное: блок «Last updated on» есть только у
  редактированных листингов. Срезаем префикс и хвост «View all revisions».
  ⚠️ текст встречается и в `<script>` (JS i18n) — узлы внутри script
  пропускать.
- **location** обязателен (на PDP стабилен; универсальный поиск «Located in:»
  по SECONDARY-span — контейнер бывает разный).

## ZIP доставки — сессионная локация (обязательна)

Без ship-to локации eBay берёт её по **IP-гео**: доставка либо не считается
(«May not ship to <страна>» — не-US IP), либо тихо считается под чужой индекс
(US-IP). Оба исхода неприемлемы, поэтому `zip` у `fetch_item` обязателен
(рекомендация — `"19701"`, единый с каталогом).

Механика (подтверждено live, июнь 2026):

- Локация **контекст-wide** (одна на сессию), живёт на стороне eBay; на
  `/itm/` URL-параметр `_stpos` **игнорируется**.
- **Сеттер — визит SRP с `_stpos={zip}`** (тот же механизм, что у каталога):
  `_nkw` = сам item_id; даже «0 results» выставляет ZIP на всю сессию.
  Кука `nonsession` тоже несёт локацию, но подписана и пишется лениво (~5с) —
  как сигнал непригодна, в коде не используется.
- **Истина — сама страница**: `"shipToLocation"` в HTML (после установки —
  `"{zip},USA"`, URL-encoded; без — IP-гео локация). Извлекает
  `html/item.py::ship_to_location`.

Поток оптимистичный: товар грузится сразу, локация сверяется по его же HTML;
мискматч → один setter-SRP → повторный заход; повторный мискматч →
`ParseError("ship_to_location")`. Стационарно — ноль лишних навигаций,
setter срабатывает один раз на сессию (замерено: 3 товара = 1 setter).

## Добыча (`EbaySession.fetch_item`)

```
fetch_item(item_id, *, zip)
goto /itm/{id} → wait_until_ready(ITEM) → main_html → ship_to_location ==
"{zip},USA"? (нет → setter-SRP → повторно) → description_html → parse
```

Готовность ITEM включает картинки карусели и `iframe#desc_ifr` как якоря
(теги в DOM), но загрузку документа фрейма ждёт отдельно `_description_html`.
Блокировка/смерть страницы лечится сессией — item повторяется ЦЕЛИКОМ на
новой странице (включая ZIP-флоу). Item-задача на завершённый листинг даст
таймаут якорей → критично (осознанно: items обрабатываются сразу после
каталога, мёртвых в потоке почти нет; увидим проблему — решим).

После каталога items автоматически НЕ парсим — только по отдельной задаче.

## Запись (`Store.apply_item` → `apply_item_snapshot`)

Payload = `ItemPage` как jsonb. БД сама: апсерт item (PDP — полный авторитет
своих полей; `image_url` SRP-миниатюры не трогает), доставка per-контекст,
EAV-дифф specifics (авторегистрация ключей), дифф галереи по md5(url),
дедуп описаний в blobs, журнал изменений триггерами. Повторная запись
идемпотентна (нулевые диффы).
